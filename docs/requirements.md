## 1. プロジェクト概要

### 1.1 目的
個人の日常タスクを視覚的に整理し、進捗状況と期限を一目で把握できるようにする、Trello風のシンプルなタスク管理Webアプリケーション「**TaskFlow**」を開発する。

### 1.2 背景
頭の中だけでタスクを管理していると、以下のような問題が発生する。

- やるべきタスクの全体像が見えず、漏れが発生する
- 何が進行中で何が完了したのかが曖昧になる
- 締め切りを忘れてしまう

これらを解決するため、ブラウザ上で手軽に使えるタスクボードを自作する。

### 1.3 学習目的（課題としての位置づけ）
本プロジェクトは Raise Tech AIエンジニアコースの課題として、以下の学習を目的とする。

- 要件定義から実装・公開までの一連のWeb開発工程を経験する
- HTML / CSS / JavaScript（バニラ）の基礎を実践的に習得する
- localStorage を用いたクライアントサイドのデータ永続化を理解する
- HTML5 Drag and Drop API を用いたインタラクションを実装する
- Git / GitHub を用いた成果物の公開と管理を行う

## 4. 前提条件・制約条件

要件を満たすうえで「成り立っているものとして扱う事柄（前提条件）」と「変更できない縛り（制約条件）」を明示する。これらが崩れた場合、仕様や動作保証の範囲が変わる。

### 4.1 前提条件

| # | 前提条件 |
|---|---|
| A-1 | 利用者は1名（開発者本人）であり、同時編集は発生しない |
| A-2 | 利用ブラウザで JavaScript が有効になっている |
| A-3 | 利用ブラウザで localStorage が利用可能（プライベートブラウジング等で無効化されていない） |
| A-4 | データは1つのブラウザ・1つの端末内でのみ利用する（端末間同期は行わない） |
| A-5 | インターネット接続は不要（初回読み込み後はオフラインでも動作する） |

### 4.2 制約条件

| # | 制約条件 | 理由 |
|---|---|---|
| C-1 | フレームワーク・外部ライブラリを使用しない（バニラJS） | 基礎技術の習得が学習目的のため |
| C-2 | サーバーサイド処理・データベースを使用しない | 静的ファイルのみで完結させるため |
| C-3 | データ保存は localStorage に限定（容量上限の目安は約5MB） | クライアントサイド永続化の習得のため |
| C-4 | スマートフォン・タブレット専用の最適化は行わない | スコープを学習範囲に限定するため |
| C-5 | 開発工数は受講スケジュールの範囲内とする | 課題としての制約 |

## 5. 機能要件

### 5.1 機能一覧（全体）

| ID | 機能名 | Phase | 優先度 |
|---|---|---|---|
| F-01 | 3つの固定リスト表示 | Phase 1 | 必須 |
| F-02 | カード追加 | Phase 1 | 必須 |
| F-03 | カード削除 | Phase 1 | 必須 |
| F-04 | カードのドラッグ＆ドロップ移動 | Phase 1 | 必須 |
| F-05 | localStorageによるデータ保存・復元 | Phase 1 | 必須 |
| F-06 | カードのタスク名編集 | Phase 2 | 必須 |
| F-07 | カードの説明文設定・編集 | Phase 2 | 必須 |
| F-08 | カードの期限（締切日）設定 | Phase 2 | 必須 |
| F-09 | 期限切れカードの赤色強調表示 | Phase 2 | 必須 |

### 5.2 各機能の詳細

#### F-01 3つの固定リスト表示
- 画面上に「未着手」「進行中」「完了」の3つのリスト（列）を横並びで表示する
- リスト名・リスト数は固定とし、ユーザーは変更・追加・削除できない

#### F-02 カード追加
- 各リストの末尾に「＋カードを追加」ボタンを配置する
- ボタン押下で入力フィールドが表示され、タスク名を入力して確定するとカードがリストに追加される
- タスク名が空の場合は追加しない

#### F-03 カード削除
- 各カードに削除ボタン（×）を表示する
- 押下時に確認ダイアログを表示し、OKで削除する

#### F-04 カードのドラッグ＆ドロップ移動
- HTML5 Drag and Drop API を用いて実装する
- リスト間の移動、および同一リスト内での並び替えを可能とする
- 外部ライブラリは使用しない

#### F-05 localStorageによるデータ保存・復元
- カード追加・編集・削除・移動のいずれの操作後も、最新状態を localStorage に保存する
- アプリ起動時に localStorage からデータを読み込み、画面に反映する
- データがない場合は、空のリスト3つを表示する（サンプルカードは表示しない）

#### F-06 カードのタスク名編集
- カードをクリックすると詳細編集UI（モーダルまたはインライン）が開き、タスク名を変更できる

#### F-07 カードの説明文設定・編集
- カード詳細編集UIで、自由記述の説明文（複数行可）を入力・編集できる
- 説明文は任意項目とする

#### F-08 カードの期限設定
- カード詳細編集UIで、期限日（年月日）を設定できる
- 期限は任意項目とする
- 日付入力には `<input type="date">` を使用する

#### F-09 期限切れカードの赤色強調表示
- カードに設定された期限が当日より過去の場合、カードの背景または枠を赤色で表示する
- 「完了」リストにあるカードは期限切れでも強調しない

### 5.3 異常系仕様

正常な操作（正常系）に対し、入力ミス・保存失敗・データ破損など「想定外だが起こりうる状況」を**異常系**と呼ぶ。これらに対するアプリの振る舞いを定義し、ユーザーがデータを失ったり操作不能になったりしないようにする。

| # | 発生条件 | 期待する挙動 |
|---|---|---|
| E-01 | カード追加・タスク名編集時に名称が空（または空白のみ） | 追加・更新を行わず、入力欄を維持する。エラーメッセージで「タスク名を入力してください」を表示 |
| E-02 | タスク名・説明文が想定文字数を大きく超える | 入力自体は許容するが、表示は省略（…）して崩れを防ぐ。最大文字数（例：タスク名100字、説明文1000字）を超える分は受け付けない |
| E-03 | 不正な日付（過去すぎる・未来すぎる・形式不正） | `<input type="date">` の制御に従う。空（未設定）に戻すことも可能とする |
| E-04 | localStorage への保存失敗（容量超過 = QuotaExceededError） | 保存に失敗した旨を通知し、直前の操作はメモリ上には反映されている状態を保つ。データの一部欠損が起きないようにする |
| E-05 | localStorage のデータが破損し JSON として解析できない（JSON.parse 失敗） | 起動時に検知し、破損データを無視して空の初期状態で起動する。ユーザーに「保存データを読み込めなかった」旨を通知する |
| E-06 | localStorage が利用不可（プライベートモード等） | 起動は可能とするが「このブラウザではデータが保存されない」旨を通知する。アプリは当該セッション内では動作する |
| E-07 | ドラッグ操作をリスト外（無効領域）でドロップ／途中でキャンセル | カードは元の位置に戻す。データは変更しない |
| E-08 | 削除確認ダイアログでキャンセルを選択 | 削除を行わず、元の状態を維持する |

## 6. 非機能要件

機能そのものではなく、「どの程度の品質・条件で動くべきか」を定める要件。

### 6.1 対応環境

| 項目 | 内容 |
|---|---|
| 対応OS | Windows / macOS（PC） |
| 対応ブラウザ | Chrome / Edge / Firefox（いずれも最新版） |
| 画面サイズ | PC画面想定（1280px以上を主たるターゲット） |
| スマホ対応 | 対応しない（スコープ外） |

### 6.2 デザイン要件
- Trello風の見た目を踏襲する
- 背景：青系のベースカラー
- カード：白背景
- 全体的にシンプル・視認性重視

### 6.3 性能要件
- 個人利用を前提とし、カード総数が数十〜数百程度の範囲で快適に動作すること
- ページ初期表示は1秒以内を目標とする
- カード操作（追加・移動・削除）後の画面反映は体感で即時（目安0.3秒以内）とする

### 6.4 データ要件・データ保全
- データはユーザー端末のブラウザ localStorage に保存する
- 外部サーバー・データベースは使用しない
- 端末・ブラウザを変えるとデータは引き継がれない（仕様として許容）
- 保存形式は単一キーのJSON文字列とする（詳細は「8. データ設計」参照）
- ブラウザのキャッシュ削除等でデータが消える可能性があることを README に明記する
- （将来拡張）データのエクスポート／インポート（JSONファイル）機能は本リリースでは扱わないが、データ構造はそれを見据えた単純な形式とする

### 6.5 ユーザビリティ要件
- 主要操作（カード追加・移動・編集・削除）は、説明書なしで直感的に行えること
- 破壊的操作（削除）には確認ステップを設けること
- 操作の結果（保存成功・失敗、エラー等）がユーザーに分かる形でフィードバックされること

### 6.6 アクセシビリティ要件（努力目標）
- 文字色と背景色のコントラストを確保し、読みやすさを担保する
- 期限切れの表現を「色のみ」に依存させず、テキストやアイコン等の補助表現も検討する（色覚特性への配慮）
- セマンティックなHTMLタグを用いる

### 6.7 セキュリティ要件
- 本アプリは個人端末内で完結し、外部送信を行わない
- ユーザー入力（タスク名・説明文）を画面に表示する際は、HTMLとして解釈させずテキストとして表示し、**XSS（クロスサイトスクリプティング：悪意あるスクリプトを埋め込まれる攻撃）** を防止する
- 認証・個人情報・機密情報は扱わない（取り扱う場合はスコープ外）

### 6.8 保守性要件
- HTML / CSS / JavaScript はファイルを役割ごとに分割する
- 変数・関数は意味の分かる命名とし、主要処理にはコメントを付す
- データ構造・主要関数の役割を README またはコメントで説明する

## 8. データ設計

### 8.1 localStorage の保存形式
- キー名：`taskflow_data`
- 値：以下構造のJSON文字列

```json
{
  "lists": [
    {
      "id": "list-todo",
      "name": "未着手",
      "cards": [
        {
          "id": "card-xxxxx",
          "title": "買い物に行く",
          "description": "牛乳、卵、パン",
          "dueDate": "2026-05-30"
        }
      ]
    },
    {
      "id": "list-doing",
      "name": "進行中",
      "cards": []
    },
    {
      "id": "list-done",
      "name": "完了",
      "cards": []
    }
  ]
}
```

### 8.2 各フィールドの仕様

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `lists[].id` | string | ○ | リストの一意ID（固定値） |
| `lists[].name` | string | ○ | リスト名（固定値） |
| `lists[].cards` | array | ○ | カードの配列 |
| `cards[].id` | string | ○ | カードの一意ID（自動採番） |
| `cards[].title` | string | ○ | タスク名 |
| `cards[].description` | string | × | 説明文（空文字許容） |
| `cards[].dueDate` | string (YYYY-MM-DD) | × | 期限日（未設定時は空文字または null） |

### 9.5 ユーザー操作フロー

代表的なユースケースについて、ユーザーが行う一連の操作手順を示す。

**フロー1：カードを追加する**
```
1. 対象リストの「＋カードを追加」を押す
2. 入力欄が表示される
3. タスク名を入力する
   ├─ 空のまま確定 → 追加されない（E-01）
   └─ 入力あり    → 4 へ
4. 確定（Enter または「追加」ボタン）
5. カードがリスト末尾に追加され、localStorage に保存される
```

**フロー2：カードを別のリストへ移動する（進捗更新）**
```
1. 対象カードをドラッグし始める
2. 移動先のリスト上へドラッグする
3. ドロップする
   ├─ 有効なリスト上     → カードが移動し保存される
   └─ 無効領域/キャンセル → 元の位置に戻る（E-07）
```

**フロー3：期限を設定し、期限切れを確認する**
```
1. カードをクリックして詳細編集UI（S-02）を開く
2. 期限欄で日付を選択する
3. 「保存」を押す → モーダルが閉じ、保存される
4. 期限が当日より前の場合、メイン画面でカードが赤色強調される
   （ただし「完了」リストのカードは強調しない：F-09）
```

**フロー4：カードを削除する**
```
1. カードの削除ボタン（×）を押す
2. 確認ダイアログが表示される
   ├─ OK       → カードが削除され保存される
   └─ キャンセル → 何もしない（E-08）
```

### 9.6 エラー・通知表示

異常系（5.3）に対応するユーザーへの表示方針。実装手段（トースト／インラインメッセージ／ダイアログ）は実装時に統一する。

| 対応する異常系 | 表示メッセージ例 | 表示方法（想定） |
|---|---|---|
| E-01 タスク名が空 | 「タスク名を入力してください」 | 入力欄付近にインライン表示 |
| E-04 保存失敗（容量超過） | 「保存に失敗しました。不要なカードを削除してください」 | 画面上部に通知（トースト） |
| E-05 データ破損 | 「保存データを読み込めなかったため、初期状態で起動しました」 | 起動時に通知 |
| E-06 localStorage 無効 | 「このブラウザではデータが保存されません」 | 起動時に通知 |

```
  ┌───────────────────────────────────────────┐
  │ ⚠ 保存に失敗しました。不要なカードを削除して │ ← 通知（トースト）
  │   ください。                          [×]  │
  └───────────────────────────────────────────┘
```

※ 詳細なワイヤーフレームは別途 `docs/wireframe.md`（または画像）で作成する。

## 10. スコープ外（本プロジェクトでは実装しないこと）

下記は意図的に対応範囲から除外する。将来の拡張余地としては残すが、本リリースでは扱わない。

- 複数ユーザー対応・チームコラボレーション機能（コメント、メンバーアサインなど）
- 複数端末間のデータ同期（クラウド保存）
- スマートフォン専用デザイン・レスポンシブ対応
- 複数ボードの作成・切り替え
- リスト自体の追加・削除・名前変更
- ラベル・色分け・優先度タグ
- 検索・フィルタ機能
- 通知機能（メール・プッシュ通知等）
- データのエクスポート／インポート（将来拡張候補）

## 11. リスク管理

プロジェクト遂行や仕様上で起こりうるリスクと、その対策を事前に整理する。**影響度**＝起きたときの大きさ、**発生可能性**＝起こりやすさ。

| # | リスク | 影響度 | 発生可能性 | 対策 |
|---|---|---|---|---|
| R-1 | ブラウザのキャッシュ削除でデータが消失する | 大 | 中 | 仕様上の制約として README に明記。将来エクスポート機能で緩和 |
| R-2 | HTML5 Drag and Drop API の実装が想定より難航する | 中 | 中 | Phase 1 の早期に技術検証（プロトタイプ）を行う |
| R-3 | localStorage の容量上限（約5MB）超過 | 中 | 低 | カード数の想定上限を設け、保存失敗時はエラー通知（E-04） |
| R-4 | ブラウザごとの D&D 挙動差で動作不一致 | 中 | 中 | 対応ブラウザ（Chrome/Edge/Firefox）で都度動作確認 |
| R-5 | ユーザー入力経由の XSS | 中 | 低 | 入力値はテキストとして表示し、HTMLとして解釈させない（6.7） |
| R-6 | 学習スケジュールの都合で工数が不足する | 中 | 中 | Phase 1（必須）を優先実装し、Phase 2 を後続に回せる構成にする |
